# Lunari Native Godot API Parity Audit

This note tracks concrete parity gaps against Godot's native C++ API surface: `ClassDB`, `Object`, engine singletons, `Variant`, and the physics/input servers that script languages ultimately call. GDScript fixtures remain useful as script-facing smoke tests, but they are secondary checks; the actual oracle is the local Godot C++ API exported by this fork. Each item should either name a passing native regression or point at the source surface that still differs.

## Validation Oracle

- Primary target: the local C++ runtime API exposed through `ClassDB`, `Variant`, native singletons, `ScriptLanguage`, and engine/scene/physics classes.
- Secondary reference: GDScript only when it is the most convenient way to exercise the same script-facing native call path or editor behavior.
- Negative diagnostics: Lunari may emit Godot's native errors when the C++ API emits them, but should not add duplicate Lunari wrapper errors for calls that returned normally.

## Fixed In This Pass

- Built-in scene scripts now follow the same core resource contract GDScript uses for editor-created built-in scripts. Lunari advertises built-in script support to the script editor, exposes the internal `script/source` property for resource serialization, and avoids external file IO when a script resource path points into a `PackedScene`.
  - GDScript reference: `modules/gdscript/gdscript.cpp`, `GDScript::_get_property_list()`, `GDScript::load_source_code()`, and `GDScriptLanguage::supports_builtin_mode()`.
  - Runtime/editor path: `lunari_script.cpp`, `LunariScript::_get_property_list()`, `ResourceFormatLoaderLunariScript::load()`, and `LunariLanguage::supports_builtin_mode()`.
  - Regression: `tests/smoke_project/check_editor_integration.gd` checks the internal `script/source` property, saves a scene with an embedded Lunari script, reloads it, verifies the source survived, and executes the built-in script after instantiation.

- Project input actions are synchronized before Lunari gameplay code runs, matching GDScript's expectation that project-defined actions such as `move_left`, `move_right`, and `jump` are already in the active `InputMap`. Lunari scripts can now use direct `Input.get_axis("move_left", "move_right")` and `Input.is_action_just_pressed("jump")` calls without script-side `InputMap.has_action` fallbacks.
  - Runtime/fallback path: `lunari_script.cpp`, `LunariScript::sync_project_input_actions()` runs during script instance creation and tree entry. The shared input sync path now calls Godot's native `InputMap::load_from_project_settings()` once before input polling, then falls back to per-action repair for project actions erased during tests or editor reloads. Runtime-added actions stay intact after later project-action syncs.
  - Expression/fallback path: `lunari_script.cpp`, direct `Input.get_axis(...)` and `Input.get_vector(...)` expression evaluation routes through native Godot `Input::get_axis(StringName, StringName)` and `Input::get_vector(StringName, StringName, StringName, StringName, float)`.
  - VM path: `lunari_vm.cpp`, bytecode/native singleton dispatch now calls the same shared `LunariScript` project-input helpers for `Input.get_axis`, `Input.get_vector`, `Input.is_action_*`, `Input.action_press/release`, and action-name `InputMap` calls before native Godot dispatch. The VM fast path accepts the optional `exact_match` argument for `Input.is_action_pressed`, `Input.is_action_just_pressed`, `Input.is_action_just_released`, `Input.get_action_strength`, and `Input.get_action_raw_strength`, matching the native overload surface instead of falling back to a one-argument-only shortcut.

  - Regression: `tests/smoke_project/check_input_get_axis_parity.gd`, `tests/smoke_project/check_input_singleton_surface_parity.gd`, `tests/smoke_project/check_engine_singleton_surface_parity.gd`, `tests/kinematic_character_project_input_sync.gd`, `tests/kinematic_character_scene_movement.gd`, and `tests/kinematic_character_gdscript_parity.gd`. The axis/vector parity check now also proves project-action sync does not wipe runtime-added actions, compares optional `exact_match` action polling/strength against GDScript, covers `Input.action_press(action, strength)` so analog/partial strengths feed `Input.get_axis(...)`, `Input.get_vector(...)`, `get_action_strength(...)`, and `get_action_raw_strength(...)` like native GDScript, covers `Input.get_vector(left, right, up, down, deadzone)` with explicit deadzone values so diagonal movement normalization and deadzone filtering stay native-compatible, and covers the native default-deadzone path where Godot averages the four action deadzones when no deadzone argument is supplied. It also verifies Godot-style `&"action"` StringName literals and `StringName` locals route through the same native input calls, and verifies `InputMap.action_add_event(...)`, `action_has_event(...)`, `action_get_events(...)`, `action_erase_event(...)`, and `action_erase_events(...)` round-trip behavior for runtime and project-synced actions. The `Input` singleton surface check compares Lunari against GDScript for `Input.use_accumulated_input`, mouse mode and cursor shape, touch/mouse emulation flags, sensor setters/getters, joypad query/vibration getters, parsed key/mouse state helpers, and `Input.flush_buffered_events()`. The `Engine` singleton surface check compares writable properties and matching getter/setter calls for `time_scale`, `max_fps`, and `physics_ticks_per_second`, verifies `Engine.get_version_info()` dictionary probing via `.has(...)`, and checks `Engine.get_architecture_name()`. The kinematic project and scene checks erase `move_left`, `move_right`, and `jump` from the active `InputMap` before instancing Lunari gameplay scripts, then require project settings to be restored before direct `Input.get_axis(...)`, `Input.is_action_just_pressed(...)`, and `InputMap.has_action(...)` calls run. The GDScript parity check then compares walking, stopping, floor collision, gravity, and jump behavior over real physics frames, covering the old duplicate `valid_condition` / VM condition-error path. Intentional missing-action return parity moved to `tests/negative/check_input_missing_action_parity.gd` so the normal input/physics pass stays free of expected native error spam; that negative fixture now compares Lunari and GDScript return values for absent actions across `Input.get_axis(...)`, `Input.get_vector(...)`, `Input.is_action_pressed(...)`, `Input.get_action_strength(...)`, and the script-bound `Input.get_action_raw_strength(...)`, while explicit output scans reject Lunari-specific wrapper noise and allow only the native Godot missing-action diagnostics.

## Known Remaining Gaps

- GDScript's analyzer/compiler/VM has validated typed opcodes and mature type adjustment for the full Variant surface. Lunari still has mixed VM and fallback execution paths, so every gameplay feature that touches native properties, arrays/dictionaries, operators, and singleton calls needs regression coverage on both paths.
  - Recent examples that required fixes: `Input.get_axis`, `Input.get_vector`, `self.velocity.x`, `self.global_position.y`, `int(self.global_position.y)`, `@GlobalScope` constants such as `KEY_LOCATION_LEFT`, and vararg global helpers such as `min(...)` / `max(...)`.
  - Fresh evidence: concrete `InputEvent.as_text()` / key text helpers are now covered in `check_input_event_surface_parity.gd`; the next native-return pass should focus on larger scene/resource chains and remaining specialized physics resources across both VM bytecode and fallback expression paths.

- Editor parity is still thinner than GDScript. Lunari has script-editor helpers, docs integration, method filtering, and completions, but not the same GDScript analyzer/LSP depth, project-wide symbol model, warning system, or inheritance-aware completion surface.
  - GDScript reference areas: `modules/gdscript/editor/`, `modules/gdscript/language_server/`, and `modules/gdscript/gdscript_analyzer.cpp`.
  - Lunari reference areas: `modules/lunari/lunari_tooling.cpp`, `lunari_analyzer.cpp`, and editor integration in `lunari_script.cpp`.

### Editor And IntelliSense Parity Backlog

GDScript's editor support is not just `complete_code()`. It has a dedicated highlighter, doc generator, language server protocol layer, workspace symbol index, native symbol database, diagnostics publisher, signature help, related-symbol resolution, rename validation, and usage search. Lunari currently exposes useful helper APIs, but they are still much thinner.

Concrete gaps to close:

- Add a Lunari language-server layer mirroring the GDScript split between extended parser, text document service, workspace service, and protocol glue.
